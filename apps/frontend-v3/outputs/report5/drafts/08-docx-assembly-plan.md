# 6. DOCX Assembly Plan

## Final DOCX Filename

`Report5_VSTEP_FrontendV3_Test_Documentation.docx`

## DOCX Structure

Keep the supplied Report 5 template style:

```txt
CAPSTONE PROJECT REPORT
Report 5 – Software Test Documentation
– Ho Chi Minh, May 2026 –
Table of Contents

I. Record of Changes
II. Testing Documentation
  1. Scope of Testing
  2. Test Strategy
    2.1 Testing Types
    2.2 Test Levels
    2.3 Supporting Tools
  3. Test Plan
    3.1 Human Resources
    3.2 Test Environment
    3.3 Test Milestones
  4. Test Cases
  5. Test Reports
```

## What Goes Into DOCX

- Summary scope tables.
- Test strategy tables.
- Test-level matrix.
- Supporting tools table.
- Human resources table.
- Test environment table.
- Test milestones table.
- Short reference to Excel test-case appendix.
- Test report summary instructions.

## What Should Not Go Into DOCX

Do not paste all detailed test cases directly into DOCX. The Report 5 template points test cases to Excel. Keeping detailed test cases in Excel makes the report cleaner and closer to the provided sample.

## Test Cases Section Text

Use this wording:

```txt
Other test case : Report5_VSTEP_FrontendV3_Test_Report.xlsx
```

If unit-test appendix is needed:

```txt
Unit Test Cases: Report5_VSTEP_FrontendV3_Unit_Test.xlsx
Other Test Cases (IT, ST, AT): Report5_VSTEP_FrontendV3_Test_Report.xlsx
```

## Final Review Checklist

- DOCX title, blank spacing, heading style, and table style match the sample.
- Excel rows stay inside the template columns and borders.
- No old sample names remain.
- Tester names match project team.
- Test cases are specific, not generic.
- Each test case has clear preconditions, steps, expected result, status, date, tester, and note.
- Scope matches `apps/frontend-v3` learner authentication, exam room, wallet, and promo-code flows.
